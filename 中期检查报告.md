## 1. 项目概述

### 1.1 项目背景

本项目基于昇腾310B（OrangePi AIpro）边缘计算平台，构建一套完整的"**检测 → 跟踪 → 云台随动**"闭环系统。USB 摄像头采集实时画面，MobileNet-SSD 轻量级模型在 NPU 上推理检测目标，DeepSORT 跟踪器维持跨帧目标 ID，提取目标中心坐标后通过 UART 串口发送至 STM32F103C8T6，STM32 通过 CAN 总线驱动两台闭环步进电机（Emm_V5.0），使摄像头云台主动跟随目标运动。

### 3.2 端到端数据流（逐帧时序）

```
第 N 帧:
  ① USB摄像头 → OpenCV V4L2 读取帧 (640×480 BGR)
  ② preprocess_frame: BGR→RGB, resize→320×320, ImageNet 归一化
  ③ NPU ACL 推理: H2D memcpy → mdl.execute → D2H memcpy (约15ms)
  ④ SSDDecoder: 3234个先验框解码 → softmax → 逐类NMS → 筛选(score≥0.35)
  ⑤ DeepSORT: 卡尔曼预测 → IOU矩阵 → 匈牙利匹配 → 中心距补充匹配 → 更新轨迹
  ⑥ 提取最大目标框中心 → 归一化至 320×240 → 串口发送 "X160Y120\n"
  ⑦ STM32 UART RX中断: 逐字节接收, '\n'触发解析
  ⑧ P控制: error_x = 160-160=0 → rpm=0 (在死区内, 不动作)
  ... (每10ms控制周期循环)
```

### 4.3 跟踪模块：简化版 DeepSORT

**Track 类（单条轨迹）：**

| 属性 | 说明 |
|------|------|
| `track_id` | 唯一递增 ID |
| `bbox` | 当前边界框 `[x1,y1,x2,y2]` |
| `kalman_filter` | 4维卡尔曼滤波器 `[x, y, vx, vy]` |
| `trail` | 最近30个中心点（绘制运动拖尾） |
| `time_since_update` | 未匹配帧计数（超 max_age 删除） |
| `hits` | 匹配次数（低于 min_hits 不显示） |

**核心平滑策略：**
```
框宽高: width  = 0.8 × old_width  + 0.2 × detection_width
分数:   score  = 0.7 × old_score  + 0.3 × detection_score
类别:   累计历史类别得分，投票决定最终类别
```

**DeepSORT 关联流程：**

```
每一帧:
  ① 所有轨迹做卡尔曼预测（匀速模型）
  ② 计算轨迹-检测 IOU 矩阵 (N×M)
  ③ 计算类别兼容掩码（同类别或任一为-1才允许匹配）
  ④ 匈牙利算法最大化 IOU 和（scipy linear_sum_assignment）
  ⑤ 阈值过滤：IOU < 0.3 的匹配丢弃
  ⑥ 未匹配的做中心距离补充匹配（归一化距离 < 1.8）
  ⑦ 匹配到的：update → 卡尔曼校正 + 平滑
  ⑧ 未匹配检测：create_track
  ⑨ 超时轨迹：删除 (time_since_update > max_age)
```

**卡尔曼滤波器参数：**

| 参数 | 值 | 说明 |
|------|-----|------|
| 状态向量 | `[x, y, vx, vy]` | 位置 + 速度 |
| dt | 1.0 | 帧间时间间隔（单位帧） |
| measurement_noise | 4.0 | 观测噪声（较大，容忍检测框抖动） |
| process_noise_position | 1.0 | 位置过程噪声 |
| process_noise_velocity | 0.25 | 速度过程噪声（较小，假设匀速） |
| initial_velocity_variance | 100.0 | 初速度方差很大（新目标方向不确定） |

### 4.4 串口输出模块：GimbalSerial

```
功能: 从检测结果中挑面积最大的目标 → 计算中心坐标 → 归一化 → UART 发送

归一化规则: nx = cx × 320 / frame_w, ny = cy × 240 / frame_h
           (固定虚拟画面 320×240，STM32 端 CENTER_X=160, CENTER_Y=120)

发送协议: "X{nx}Y{ny}\n" （ASCII 编码）
发送频率: 限频 20Hz（默认）
安全机制: 无目标 → 不发送 → STM32 端 200ms 看门狗自动急刹
```

### 5.3 主控制逻辑（`main.c`）

**关键全局参数：**

```c

char    rx_buffer[32];        // 接收缓冲
char    process_buffer[32];   // 解析缓冲（双缓冲防竞争）

// === P 控制器参数 ===
#define CENTER_X  160         // 虚拟画面中心 X (320/2)
#define CENTER_Y  120         // 虚拟画面中心 Y (240/2)

```

**主循环三部曲：**

```
┌──────────────────────────────────────────────────────┐
│ ① 坐标解析（每收到一帧执行一次）                      │
│                                                      │
│   识别帧头 'X' → 用 strchr 找 'Y' → atoi 解析坐标    │
│   target_x, target_y → target_valid=1 → 喂狗         │
├──────────────────────────────────────────────────────┤
│ ② P 控制器（每 10ms 执行一次）                        │
│                                                      │
│   error_x = target_x - 160     // 像素偏差           │
│   error_y = target_y - 120                           │
│                                                      │
│   if |error_x| > 25:                                 │
│       rpm_x = |error_x| × 0.1   // 比例控制          │
│       rpm_x = clamp(rpm_x, 0, 150)                   │
│       dir_x = (error_x > 0) ? CW : CCW               │
│                                                      │
│   Vel_Control(1, dir_x, rpm_x, 15, 0)  // CAN 发送   │
│   HAL_Delay(1)  ← 错峰，清空 CAN 邮箱                │
│   Vel_Control(2, dir_y, rpm_y, 15, 0)  // CAN 发送   │
├──────────────────────────────────────────────────────┤
│ ③ 看门狗（200ms 超时）                               │
│                                                      │
│   if 距上次收数据 > 200ms:                           │
│       Vel_Control(1, 0, 0, ...)  // 电机1急刹        │
│       Vel_Control(2, 0, 0, ...)  // 电机2急刹        │
└──────────────────────────────────────────────────────┘
```

**为什么使用速度环而非位置环？**
- 实时追踪场景中目标持续移动，位置环的"到达→停止→再追"模式会产生锯齿运动
- 速度环让电机始终平滑追赶，画面不抖动
- 死区（DEADZONE=25）防止目标在中心附近时电机来回振荡

**为什么两个电机命令间要 HAL_Delay(1)？**
- STM32F103 的 bxCAN 只有 3 个发送邮箱，快速连续发送可能填满
- 1ms 间隔确保上一帧已从邮箱发出，避免 HAL_CAN_AddTxMessage 返回 BUSY

### 5.4 UART 中断接收设计

```c
// 单字节中断链式接收（永不停止）
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    if (rx_byte == '\n') {
        // 帧结束：拷贝到 process_buffer（双缓冲），设置 rx_complete=1
        strcpy(process_buffer, rx_buffer);
        rx_complete = 1;
        rx_index = 0;
    } else if (rx_byte != '\r') {
        // 忽略回车，追加数据
        rx_buffer[rx_index++] = rx_byte;
    }
    HAL_UART_Receive_IT(&huart1, &rx_byte, 1);  // 链式重挂
}
```

**双缓冲设计**：`rx_buffer` 用于实时接收，`process_buffer` 供主循环解析。帧结束时 `strcpy` 拷贝，避免主循环解析过程中被新数据覆盖。

### 5.5 CAN 通信层

**波特率**：72MHz / 12(Prescaler) / (1+4+1) = **500kbps**

**扩展帧 ID 编码**：
```
ExtId = (电机地址 << 8) | 分包序号
       └── 高8位 ──┘   └─ 低8位 ──┘
```

**`can_SendCmd()` 自动分包算法**：
```
cmd[0]=地址, cmd[1]=功能码, cmd[2..len-1]=数据, cmd[len-1]=0x6B(校验)

子帧格式: [功能码][数据0..数据6]
每子帧 DLC ≤ 8（1 字节功能码 + 最多 7 字节数据）
分包序号 packNum 从 0 递增
```

**CAN 应答风暴规避**：CAN RX 中断虽已注册 NVIC，但主循环中 `HAL_CAN_ActivateNotification` 被有意注释掉。原因是每个电机命令都会回一帧应答，若开 RX 中断又不及时读 FIFO，应答帧会触发中断风暴（优先级高于 SysTick），导致 `HAL_Delay` 卡死，第二条命令永远发不出去。

### 5.6 电机驱动器协议（Emm_V5.0）

Emm_V5.0 是"张大头闭环步进伺服"驱动器，通过 CAN 总线通信。本项目主要使用以下 API：

| API | 功能码 | 用途 |
|-----|--------|------|
| `Emm_V5_Vel_Control()` | 0xF6 | ★ 速度模式（主用）：`[addr,0xF6,dir,vel_H,vel_L,acc,snF,0x6B]` |
| `Emm_V5_En_Control()` | 0xF3 | 电机使能/关闭 |
| `Emm_V5_Stop_Now()` | 0xFE | 立即停止（看门狗急刹） |
| `Emm_V5_Synchronous_motion()` | 0xFF | 多电机同步启动 |

速度模式 CAN 帧格式（8字节）：
```
Byte[0] = 电机地址  Byte[1] = 0xF6   Byte[2] = 方向    Byte[3] = vel高
Byte[4] = vel低     Byte[5] = acc     Byte[6] = 同步标志 Byte[7] = 0x6B
```

## 8. 关键技术决策与调优

### 8.1 为什么选择速度环而非位置环

| 维度 | 速度环 | 位置环 |
|------|--------|--------|
| 追踪平滑度 | 连续变速，画面不抖动 | 到达-停-再追，锯齿运动 |
| 目标持续移动 | ✅ 始终保持追赶 | ❌ 永远在"到达-再动"循环 |
| 目标静止 | 接近中心时停转 | 精确定位 |
| 适合场景 | 实时追踪 | 点到点定位 |

**结论**：实时追踪场景选择速度环，用死区防止微抖。

### 8.3 CAN 应答风暴问题的发现与解决

**问题**：初始化时尝试开启 CAN RX 中断（`HAL_CAN_ActivateNotification`），结果系统卡死在初始化阶段。

**原因分析**：每个电机命令都会回一帧应答。若开启 RX 中断，第一个命令发出后立即触发中断 → 进入 ISR 读 FIFO → 又有新应答 → 中断嵌套 → 中断风暴。CAN RX 中断优先级高于 SysTick，导致 `HAL_Delay()` 永远等不到 `HAL_GetTick()` 更新，系统假死。

**解决方案**：注释掉 `HAL_CAN_ActivateNotification`，本场景只需发送无需接收。CAN RX 中断 NVIC 虽仍初始化为避免 HAL 报错，但实际不会触发回调。

## 9. 遇到的问题与解决方案

| 问题 | 影响 | 解决方案 |
|------|------|---------|
| **CAN 应答中断风暴** | 系统卡死，第二条命令发不出 | 关闭 CAN RX 中断，只发不收 |
