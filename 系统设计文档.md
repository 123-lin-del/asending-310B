## 1. 系统概述

### 1.1 项目背景

随着边缘智能计算的发展，在嵌入式设备上实现实时目标检测与跟踪已成为计算机视觉领域的重要应用方向。本系统基于昇腾310B（OrangePi AIpro）开发板，构建一套完整的"**检测—跟踪—云台随动**"闭环系统，使摄像头能够主动跟随目标运动。

### 1.2 系统目标

系统的核心工作流程为：

1. **摄像头捕捉**：USB 摄像头实时采集视频画面
2. **目标检测**：MobileNet-SSD 轻量级模型在 NPU 上进行推理，检测画面中的人物并输出边界框坐标
3. **目标跟踪**：DeepSORT 跟踪器（卡尔曼滤波 + 匈牙利算法）对检测目标分配稳定 ID，在连续帧中维持轨迹
4. **坐标提取**：从跟踪目标的边界框中提取中心点坐标 `(cx, cy)`
5. **串口传输**：将中心坐标通过 UART 串口发送至 STM32F103C8T6
6. **云台控制**：STM32 根据坐标偏差计算 PID 控制量，驱动 SG90 舵机云台转动，使摄像头始终跟随目标运动

### 1.3 开发性质

本项目为**二次开发**：CANN 8.0 开发环境已搭建完成，基础检测与跟踪框架代码由老师提供并烧录至开发板。团队工作重心在于：理解现有代码结构和算法原理、调优检测与跟踪参数、完成 STM32 云台控制开发，以及全链路联调优化。

本设计文档的技术方案基于以下参考资料：

## 4. 检测模块设计

### 4.1 MobileNet-SSD 模型结构

#### 4.1.1 MobileNet 骨干网络

MobileNet 的核心设计思想是用**深度可分离卷积**（Depthwise Separable Convolution）替代标准卷积，将标准卷积拆分为两步：

1. **Depthwise Convolution**：对每个输入通道分别做空间卷积（逐通道）
2. **Pointwise Convolution**：使用 1×1 卷积完成通道间的信息融合

当卷积核大小 $K=3$ 时，计算量从标准卷积的 $D^2 \cdot K^2 \cdot M \cdot N$ 降至 $D^2 \cdot K^2 \cdot M + D^2 \cdot M \cdot N$，大幅降低边缘设备的计算负担。

MobileNet 由以下几类模块反复堆叠而成：

- 普通卷积层：最初级特征提取
- 深度卷积层：逐通道空间信息提取
- 逐点卷积层：跨通道特征融合
- 下采样层：逐步扩大感受野

本案例支持 MobileNet v1 ~ v4 四个版本，演进主线为：

| 版本 | 核心贡献 | 适用场景 |
|------|---------|---------|
| **v1** | 系统化深度可分离卷积 | 最基础结构，入门首选 |
| **v2** | Inverted Residual + Linear Bottleneck | 轻量下保留表达能力 |
| **v3** | SE 注意力 + h-swish + NAS 搜索 | 硬件效率与精度平衡 |
| **v4** | 硬件感知的模块组合优化 | 面向实际部署的综合收益 |

本项目默认使用 **MobileNet v2**，在轻量化与检测精度间取得最佳平衡。

#### 4.1.2 SSD 检测头

SSD（Single Shot MultiBox Detector）作为一阶段检测器，核心思想是在**多个尺度特征图**上通过预定义的先验框（Default Boxes / Prior Boxes）直接回归目标位置和类别，无需候选区域生成阶段。

SSD 关键设计要素：

- **多尺度检测**：在 6 个不同分辨率特征图上分别预测，兼顾大目标与小目标
- **先验框机制**：每个特征图位置预定义一组不同长宽比的 default boxes（如 1:1, 1:2, 2:1, 1:3, 3:1）
- **双任务头**：分类分支（各类别置信度）+ 回归分支（位置偏移量）
- **解码与 NMS**：将网络输出的偏移量反变换为实际边界框坐标，再通过非极大值抑制去除重复框

### 4.2 预处理规范

模型的输入预处理流程必须与训练阶段的约定严格一致，任何偏差都会导致推理结果发生明显偏移。具体步骤如下：

```
输入：原始帧 (H × W × 3, BGR, uint8)
            │
            ▼
   ① BGR → RGB 色彩空间转换
            │
            ▼
   ② Resize 到 320 × 320
            │
            ▼
   ③ 像素值归一化 [0, 255] → [0.0, 1.0]（除以 255）
            │
            ▼
   ④ ImageNet 标准化
      R: (R - 0.485) / 0.229
      G: (G - 0.456) / 0.224
      B: (B - 0.406) / 0.225
            │
            ▼
   ⑤ HWC → CHW 通道重排 (320, 320, 3) → (3, 320, 320)
            │
            ▼
   ⑥ 增加 batch 维度 (3, 320, 320) → (1, 3, 320, 320)
            │
            ▼
输出：输入张量 (1 × 3 × 320 × 320, float32, NCHW)
```

### 4.4 解码与 NMS 后处理

#### 4.4.1 解码原理

网络输出的并非最终边界框坐标，而是相对于 default boxes 的偏移量：

- **位置解码**：利用 `scale_xy` 和 `scale_wh` 对偏移量进行反变换

  $$cx = prior\\_cx + offset\\_cx \times scale\\_xy \times prior\\_w$$
  $$cy = prior\\_cy + offset\\_cy \times scale\\_xy \times prior\\_h$$
  $$w = prior\\_w \times \exp(offset\\_w \times scale\\_wh)$$
  $$h = prior\\_h \times \exp(offset\\_h \times scale\\_wh)$$

- **坐标转换**：将中心点形式 `(cx, cy, w, h)` 转换为左上角/右下角形式 `(x1, y1, x2, y2)`
- **类别筛选**：对每个 prior 选出最佳前景类别，筛掉置信度低于 `score_threshold` 的候选框

#### 4.4.2 优化后的解码策略

当前工程版本（`ssdlite/decoder.py`）对解码流程做了面向实时性的优化：

- 对每个 prior 先在候选前景类别中选出一个最佳类别，而非"每个类别都扫全部 prior"
- 只保留分数超过阈值的候选框
- 再按类别分别做 NMS

这在大幅减少无效类别扫描的同时，避免同一个 prior 在多个类别上重复进入 NMS。

#### 4.4.3 NMS（非极大值抑制）

NMS 的核心作用是**去重**：当多个候选框指向同一目标时，只保留最可信的框。

NMS 处理逻辑：

```
输入：候选框列表 boxes, 置信度 scores, IOU 阈值
输出：保留框的索引

1. 按置信度从高到低排序候选框
2. while 候选框列表非空:
   a. 取出当前最高分框，标记为保留
   b. 计算其余框与该保留框的 IOU
   c. 删除所有 IOU > threshold 的框
3. 返回所有保留框的索引
```

**NMS 对跟踪的关键意义**：如果检测器把同一目标输出成多个高重叠框，跟踪模块在创建轨迹时可能误判为多个独立目标，造成重复轨迹和 ID 混乱。因此 NMS 是整个"检测到跟踪"流水线稳定运行的重要保障。

#### 4.4.4 NMS 阈值选择权衡

| IOU 阈值 | 去重强度 | 相邻目标保留能力 | 适用场景 |
|----------|---------|----------------|---------|
| 0.3（严格） | 强 | 弱，可能误删相邻目标 | 稀疏场景 |
| **0.5（默认）** | **适中** | **适中** | **通用场景** |
| 0.7（宽松） | 弱 | 强 | 密集场景（人群） |

### 4.5 性能拆分

## 5. 跟踪模块设计

### 5.1 DeepSORT 算法概述

DeepSORT 在 SORT（Simple Online and Realtime Tracking）基础上引入外观特征建模。本案例采用简化版本，保留四个核心机制：

1. **卡尔曼滤波**：对每条轨迹维护状态估计，通过 Predict-Update 两步平滑观测噪声
2. **匈牙利算法**：基于 IOU 矩阵实现检测与轨迹的全局最优一一匹配
3. **IOU 匹配 + 类别约束**：以交并比为主要几何相似度，叠加类别兼容矩阵避免跨类别误匹配
4. **轨迹生命周期管理**：通过 `max_age` 和 `min_hits` 控制轨迹的创建、保留与删除

### 5.2 系统状态模型

#### 5.2.1 轨迹状态表示

每条轨迹（`Track` 对象）维护以下状态：

| 属性 | 类型 | 含义 |
|------|------|------|
| `track_id` | int | 目标唯一编号 |
| `bbox` | `[x1,y1,x2,y2]` | 当前边界框 |
| `score` | float | 当前检测置信度（经平滑） |
| `class_id` | int | 目标类别 |
| `class_scores` | dict | 历史类别分数累计（用于稳定分类） |
| `trail` | list | 历史中心点序列（用于轨迹拖尾可视化） |
| `kalman_filter` | KalmanFilter | 卡尔曼滤波器（运动状态估计） |
| `time_since_update` | int | 距上次匹配成功的帧数 |
| `hits` | int | 累计成功匹配次数 |

#### 5.2.2 卡尔曼滤波状态模型

采用简化线性状态向量：

$$x = [cx, cy, v_x, v_y]^T$$

其中 $(cx, cy)$ 为目标中心位置，$(v_x, v_y)$ 为速度分量。

**状态转移矩阵**（匀速运动模型）：

$$F = \begin{bmatrix} 1 & 0 & 1 & 0 \\ 0 & 1 & 0 & 1 \\ 0 & 0 & 1 & 0 \\ 0 & 0 & 0 & 1 \end{bmatrix}$$

**观测矩阵**（只能观测位置，不能直接观测速度）：

$$H = \begin{bmatrix} 1 & 0 & 0 & 0 \\ 0 & 1 & 0 & 0 \end{bmatrix}$$

**过程噪声与观测噪声**（工程版设置）：

- 位置过程噪声：较小（假设目标运动较平滑）
- 速度过程噪声：适中（允许速度有一定变化）
- 观测噪声：根据检测器可靠性设定（检测框的定位噪声）

### 5.3 卡尔曼滤波两阶段

#### 5.3.1 Predict（预测阶段）

```python
def predict(self):
    self.x = self.F @ self.x          # 状态外推
    self.P = self.F @ self.P @ self.F.T + self.Q  # 协方差传播
    return self.x
```

**作用**：即使当前帧还没有拿到新的检测结果，也先根据上一时刻状态预测目标现在大概会出现在哪里。这使跟踪器能在短时遮挡或漏检时维持轨迹连续。

#### 5.3.2 Update（更新阶段）

```python
def update(self, z):
    measurement = np.asarray(z, dtype=np.float32).reshape((2, 1))
    innovation = measurement - self.H @ self.x                     # 残差
    innovation_covariance = self.H @ self.P @ self.H.T + self.R    # 残差协方差
    kalman_gain = np.linalg.solve(innovation_covariance.T,         # 卡尔曼增益
                                   (self.P @ self.H.T).T).T
    self.x = self.x + kalman_gain @ innovation                      # 状态修正
    # Joseph form 协方差更新（数值更稳定）
    correction = self._identity - kalman_gain @ self.H
    self.P = correction @ self.P @ correction.T \
             + kalman_gain @ self.R @ kalman_gain.T
```

**作用**：一旦当前帧有新的检测框匹配到这条轨迹，就用新的观测值修正状态估计。

**卡尔曼增益的直观意义**：

- 如果观测噪声很大 → 卡尔曼增益变小 → 更信任模型预测
- 如果观测噪声较小 → 卡尔曼增益变大 → 更信任当前检测结果

### 5.4 数据关联

#### 5.4.1 IOU 矩阵与类别约束

在多目标匹配前，系统先构造两个矩阵：

1. **IOU 矩阵**：每条轨迹预测框与每个检测框的交并比
2. **类别兼容矩阵**：轨迹类别与检测类别是否一致

两者叠加形成代价矩阵：

$$cost_{ij} = \begin{cases} IOU(trk_i, det_j) & \text{if } class(trk_i) = class(det_j) \\ -1.0 & \text{otherwise} \end{cases}$$

不兼容的类别对在进入匈牙利匹配前就被屏蔽，避免"行人轨迹误匹配到车辆框"等明显错误。

#### 5.4.2 匈牙利算法全局匹配

匈牙利算法（Kuhn-Munkres Algorithm）解决的是二分图上的**全局最优一一匹配**问题：

- 将"轨迹"和"检测框"分别视为两侧节点
- 将 IOU 相似度矩阵视为边权
- 求解全局总相似度最大（或总代价最小）的一一配对方案

本案例通过 `scipy.optimize.linear_sum_assignment(-cost_matrix)` 调用标准实现。由于该函数默认求解最小化问题，而此处构造的是 IOU 相似度矩阵（越大越优），因此对代价矩阵取负号将"最大化相似度"转化为"最小化负相似度"。

**匈牙利算法 vs 贪心匹配**：

| 策略 | 优点 | 局限 |
|------|------|------|
| 贪心匹配（取最大 IOU 配对） | 简单快速 | 局部最优可能破坏全局最优，密集场景易出错 |
| **匈牙利算法（全局最优）** | 全局最优，避免多目标冲突 | 计算开销略高 |

#### 5.4.3 中心距离补充匹配

在 IOU 关联失败后，系统增加一轮基于中心距离的补充匹配：

- 计算轨迹中心与检测框中心的归一化距离
- 结合类别约束，对小于 `center_distance_threshold` 的候选再做一次匈牙利关联
- 适用于快速运动目标、短时失配恢复等场景

### 5.5 DeepSORT.update 核心流程

```python
def update(self, detections):
    # 步骤1：对所有已有轨迹做卡尔曼预测
    for track in self.tracks:
        track.predict()

    # 步骤2：构造 IOU 矩阵 + 类别兼容矩阵，匈牙利算法求全局最优匹配
    matched, unmatched_detections, unmatched_tracks = self._associate(detections)

    # 步骤3：用匹配到的检测结果更新对应轨迹（卡尔曼 update + 平滑）
    for track_idx, detection_idx in matched:
        self.tracks[track_idx].update(detections[detection_idx])

    # 步骤4：对未匹配的检测创建新轨迹（初始状态 hits=1）
    for detection_idx in unmatched_detections:
        self._create_track(detections[detection_idx])

    # 步骤5：删除长期未更新的轨迹（time_since_update > max_age）
    self.tracks = [t for t in self.tracks
                   if t.time_since_update <= self.max_age]
```

这五步完整体现了多目标跟踪的标准流程：**预测 → 匹配 → 更新 → 创建 → 清理**。

### 5.6 轨迹平滑

当前工程版本对轨迹框尺寸和置信度分数做了指数平滑：

$$size_t = \alpha \cdot size_{det} + (1 - \alpha) \cdot size_{t-1}$$
$$score_t = \beta \cdot score_{det} + (1 - \beta) \cdot score_{t-1}$$

| 参数 | 作用 | 调节方向 |
|------|------|---------|
| `size_smoothing` | 框宽高变化平滑度 | 增大 → 框抖动更小但尺寸响应更慢 |
| `score_smoothing` | 置信度显示平滑度 | 增大 → 显示分数更稳定但对置信度突变更迟钝 |

### 5.7 轨迹生命周期参数

| 参数 | 默认值 | 调优方向 | 对跟踪效果的影响 |
|------|--------|---------|----------------|
| `max_age` | 30 | 增大（60→120）| 遮挡恢复能力增强，但旧轨迹清理变慢 |
| `min_hits` | 3 | 增大（5→7）| 抑制误检，但新轨迹出现变慢 |
| `iou_threshold` | 0.3 | 增大（0.4→0.5）| 匹配更保守，快速运动易失配；减小则邻近日标易错配 |
| `center_distance_threshold` | 1.4 | 增大（2.0）| 补充匹配更积极，但可能增加误匹配 |

### 6.1 物理层参数

| 参数 | 设定值 | 说明 |
|------|--------|------|
| 接口 | UART-TTL | 昇腾端通过 CH340G USB 转 TTL 输出 |
| 波特率 | **115200 bps** | 兼顾速度与可靠性 |
| 数据位 | 8 bit | 标准配置 |
| 停止位 | 1 bit | 标准配置 |
| 校验位 | 无 | 由应用层校验和保证完整性 |
| 流控制 | 无 | 简化硬件连接 |

### 6.2 帧格式

采用 7 字节定长帧，帧头帧尾双校验设计：

```
┌────────┬─────────┬─────────┬─────────┬─────────┬────────┬────────┐
│ Byte 0 │ Byte 1  │ Byte 2  │ Byte 3  │ Byte 4  │ Byte 5 │ Byte 6 │
├────────┼─────────┼─────────┼─────────┼─────────┼────────┼────────┤
│ 帧头   │ cx 高位  │ cx 低位  │ cy 高位  │ cy 低位  │ 校验和  │ 帧尾   │
│ 0xAA   │ (bit15-8)│ (bit7-0)│ (bit15-8)│ (bit7-0)│ XOR    │ 0x55   │
└────────┴─────────┴─────────┴─────────┴─────────┴────────┴────────┘
```

**校验和计算**：

$$checksum = 0xAA \oplus cx\_high \oplus cx\_low \oplus cy\_high \oplus cy\_low$$

### 6.3 数据范围

| 字段 | 类型 | 范围 | 说明 |
|------|------|------|------|
| `cx` | uint16 (大端) | 0 ~ 640 | 目标中心 X 坐标（对应 640×480 分辨率） |
| `cy` | uint16 (大端) | 0 ~ 480 | 目标中心 Y 坐标 |
| 校验和 | uint8 | 0 ~ 255 | 前 5 字节的异或值 |

> **注意**：当无有效跟踪目标时，发送 `cx=0, cy=0` 作为"丢失目标"信号，STM32 端接收到后停止舵机运动或执行回中操作。

### 6.4 STM32 接收状态机

```
                    ┌──────────────────┐
                    │   WAIT_HEADER    │  等待 0xAA

                             │ 收到 0xAA
                             ▼
                    ┌──────────────────┐
                    │   READ_DATA      │  连续接收 4 字节数据
                    │ idx = 0..3       │  {cx_high, cx_low, cy_high, cy_low}
                    └────────┬─────────┘
                             │ 收到 4 字节

                    ┌──────────────────┐
                    │   READ_CHECKSUM  │  接收校验和
```

### 6.5 昇腾端发送逻辑

```python
def send_coordinates(ser, cx, cy):
    """通过串口发送目标中心坐标"""
    frame = bytearray(7)
    frame[0] = 0xAA                          # 帧头
    frame[1] = (cx >> 8) & 0xFF              # cx 高字节
    frame[2] = cx & 0xFF                     # cx 低字节
    frame[3] = (cy >> 8) & 0xFF              # cy 高字节
    frame[4] = cy & 0xFF                     # cy 低字节
    frame[5] = frame[0] ^ frame[1] ^ frame[2] ^ frame[3] ^ frame[4]  # 校验和
    frame[6] = 0x55                          # 帧尾
    ser.write(frame)
```

### 7.2 坐标映射关系


### 7.3 PID 控制算法

#### 7.3.1 二轴独立 PID

水平轴和俯仰轴各有一个独立的 PID 控制器：

**水平轴（X 轴）**：

$$error_x(n) = cx(n) - 320$$
$$u_x(n) = Kp_x \cdot error_x(n) + Ki_x \cdot \sum_{k=0}^{n} error_x(k) + Kd_x \cdot [error_x(n) - error_x(n-1)]$$
$$pwm_x(n) = pwm_x(n-1) + u_x(n)$$

**俯仰轴（Y 轴）**：

$$error_y(n) = cy(n) - 240$$
$$u_y(n) = Kp_y \cdot error_y(n) + Ki_y \cdot \sum_{k=0}^{n} error_y(k) + Kd_y \cdot [error_y(n) - error_y(n-1)]$$
$$pwm_y(n) = pwm_y(n-1) + u_y(n)$$

#### 7.3.2 三项作用总结

| 分量 | 公式 | 作用 | 物理直觉 |
|------|------|------|---------|
| **P（比例）** | $Kp \cdot error$ | 与当前偏差成正比 | 偏差越大，转得越快 |
| **I（积分）** | $Ki \cdot \sum error$ | 累积历史偏差 | 目标静止但没对准时持续微调 |
| **D（微分）** | $Kd \cdot \Delta error$ | 反映偏差变化率 | 目标快到时提前减速，抑制过冲 |

#### 7.3.3 PID 参数整定策略

从保守参数起步，逐步整定：

| 阶段 | Kp | Ki | Kd | 目标 |
|------|----|----|----|------|
| 起步 | 0.05 | 0 | 0 | 确认方向正确、无振荡 |
| 粗调 | 逐步增大 → | 0 | 0 | 提高响应速度，至轻微振荡 |
| 精调 | 0.8×临界 | 0.01~0.05 | 0.1~0.5 | 消除稳态误差，抑制过冲 |

#### 7.3.4 工程优化措施

| 优化项 | 方法 | 作用 |
|--------|------|------|
| **死区控制** | `if abs(error) < 阈值: u = 0` | 防止目标接近中心时舵机微颤 |
| **输出限幅** | `pwm = clamp(pwm, min, max)` | 防止舵机超出 0°~180° 范围 |
| **低通滤波** | `pwm = 0.85×pwm + 0.15×pwm_old` | 平滑 PWM 输出，减少抖动 |
| **积分分离** | 大偏差时 Ki=0 | 防止积分饱和导致的超调 |

### 7.4 PWM 信号生成

STM32 通过定时器 TIM2 产生两路 50Hz PWM 信号：

| 参数 | 设定值 | 说明 |
|------|--------|------|
| TIM 时钟 | 72 MHz | APB1 总线时钟 |
| 预分频器 (PSC) | 144 - 1 | 72 MHz / 144 = 500 kHz |
| 自动重装载 (ARR) | 10000 - 1 | 500 kHz / 10000 = 50 Hz |
| 占空比范围 | 250 ~ 1250 | 0.5ms ~ 2.5ms (20ms 周期的 2.5%~12.5%) |

```
0°   → 脉宽 0.5ms  → CCR = 250
90°  → 脉宽 1.5ms  → CCR = 750
180° → 脉宽 2.5ms  → CCR = 1250
```

### 7.5 STM32 主循环流程

```c
// 伪代码：STM32 主循环
volatile uint16_t target_cx = 0, target_cy = 0;
volatile uint8_t target_valid = 0;  // 目标有效标志

int main(void) {
    HAL_Init();
    SystemClock_Config();
    MX_USART1_UART_Init();   // 115200, 中断接收
    MX_TIM2_Init();          // 50Hz PWM, CH1(PA0) + CH2(PA1)

    // 舵机初始位置：对准画面正向
    uint16_t pwm_x = 750;  // 90° (居中)
    uint16_t pwm_y = 750;  // 90° (居中)

    while (1) {
        if (target_valid) {
            // 计算偏差
            int16_t error_x = (int16_t)target_cx - 320;
            int16_t error_y = (int16_t)target_cy - 240;

            // PID 计算（二轴独立）
            int16_t u_x = PID_Compute(&pid_x, error_x);
            int16_t u_y = PID_Compute(&pid_y, error_y);

            // 更新 PWM，带输出限幅
            pwm_x = CLAMP(pwm_x + u_x, 250, 1250);
            pwm_y = CLAMP(pwm_y + u_y, 250, 1250);

            target_valid = 0;  // 清除标志，等待下一帧坐标
        }

        // 死区控制：偏差很小时不更新 PWM（减轻舵机抖动）
        // 在 PID_Compute 内部实现

        TIM2->CCR1 = pwm_x;  // 水平舵机
        TIM2->CCR2 = pwm_y;  // 俯仰舵机

        HAL_Delay(20);  // 约 50Hz 控制频率
    }
}
```

```python
import serial

        cx = int((x1 + x2) / 2)
```

## 附录 A. 编码规范

### 昇腾端（Python）

- 遵循 PEP 8 编码规范
- 关键函数添加 docstring 注释
- 模型路径、串口设备路径通过命令行参数传入，不硬编码
- 异常处理：串口断开、摄像头断开、推理异常时优雅退出

### STM32 端（C）

- 遵循 MISRA C 子集（安全关键代码规范）
- 中断服务函数尽量简短，仅做数据接收和标志位设置
- PID 计算放在主循环中，避免中断内浮点运算
- 关键参数通过 `#define` 宏定义，便于调整

## 附录 C. 故障排查指南

| 故障现象 | 可能原因 | 排查步骤 |
|---------|---------|---------|
| NPU 推理失败 | CANN 环境未正确加载 | `source /usr/local/Ascend/ascend-toolkit/set_env.sh` 后重试 |
| 摄像头无法打开 | 设备节点权限不足或设备号不对 | `ls /dev/video*` 确认设备号，`sudo chmod 666 /dev/video0` |
| 跟踪 ID 频繁跳变 | IOU 阈值过小或 max_age 过小 | 增大 `--track-iou-threshold` 和 `--track-max-age` |
| 串口无法通信 | 端口不对或权限不足 | `ls /dev/ttyUSB*` 确认端口，`sudo chmod 666 /dev/ttyUSB0` |
| 舵机不转 | 供电不足或 PWM 信号异常 | 舵机独立供电（5V/2A），示波器检查 PA0/PA1 波形 |
| 云台振荡 | PID 参数过大 | 降低 Kp 值（从 0.05 起步），增加死区阈值 |
| 舵机抖动 | PWM 信号波动 | 增加低通滤波系数，检查电源纹波 |
| FPS 偏低 | Decode 耗时过长 | 减小 `--max-detections`，使用 `--track-classes person` |
