## 1 背景与问题定义

传统的固定摄像头只能覆盖有限视角，当目标在画面中移动甚至离开中心区域时，画面信息便失去了有效利用价值。本项目的核心问题是：**如何让摄像头"看见"目标后主动跟随它运动？**

要回答这个问题，系统需要串联三项能力：

1. **看见**——在每一帧画面里找到目标（检测）
2. **认出**——跨帧维持对同一目标的认识（跟踪）
3. **跟随**——把目标位置信息传递给机械结构，驱动摄像头转动（云台控制）

这三项能力对应本项目的三段技术链路：MobileNet-SSD 实时检测 → DeepSORT 多目标跟踪 → STM32 云台 PID 闭环控制。整套系统运行在昇腾310B（OrangePi AIpro）边缘计算平台上，通过串口与 STM32F103C8T6 协同工作。

### 1.1 为什么选择这条技术路线

昇腾310B 面向边缘智能计算场景，算力有限但实时性要求高。在这类约束下，直接使用参数量庞大的检测器（如 Faster R-CNN、YOLOv8-Large）会同时带来理解成本和工程调试成本的上升，系统主线反而不易把握。

MobileNet-SSD 的优势不在于"它是最强的检测前端"，而在于它能以较低的复杂度把"检测到跟踪"这条主线讲清楚：

- MobileNet 的深度可分离卷积（Depthwise + Pointwise）将标准卷积拆成两步，计算量从 $D^2 \cdot K^2 \cdot M \cdot N$ 降至 $D^2 \cdot K^2 \cdot M + D^2 \cdot M \cdot N$，当 $K=3$ 时节省显著
- SSD 作为一阶段检测器，在多尺度特征图上通过先验框直接回归位置与类别，无需候选区域生成，推理速度快
- DeepSORT 在 SORT 基础上引入外观特征建模，本案例保留其最核心的卡尔曼滤波预测 + 匈牙利算法全局匹配机制，难度适中，适合入门

## 2 系统总体设计

### 2.1 架构总览

系统由两块处理器协同完成闭环控制：

```
                    ┌─────────────────────────────────────┐
                    │        OrangePi AIpro (310B)        │
                    │                                     │
 USB 摄像头 ──────▶│  图像采集 → 预处理 → NPU推理 → 解码  │
                    │      → NMS → DeepSORT跟踪           │
                    │          → 提取目标中心坐标           │
                    │              → UART发送              │
                    └──────────────────┬──────────────────┘
                                       │ 串口 115200bps
                    ┌──────────────────▼──────────────────┐
                    │         STM32F103C8T6               │
                    │                                     │
                    │  串口接收 → 解析坐标 → PID计算        │
                    │          → PWM输出 → 舵机驱动         │
                    └──────────────────┬──────────────────┘
                                       │
                    ┌──────────────────▼──────────────────┐
                    │        二轴 SG90 舵机云台            │
                    │    (水平轴 + 俯仰轴，摄像头安装其上)   │
                    └─────────────────────────────────────┘
```

### 2.2 数据流与关键接口

整条数据通路可分为六个环节，每个环节的输入输出关系如下：

| 环节 | 模块 | 输入 | 输出 | 关键参数 |
|------|------|------|------|---------|
| 图像采集 | OpenCV VideoCapture | 摄像头视频流 | 原始帧 (640×480 BGR) | 帧率 30fps |
| 预处理 | preprocess_frame | 原始帧 | NCHW 张量 (1×3×320×320) | BGR→RGB, 归一化, ImageNet 标准化 |
| NPU 推理 | backend_base | 输入张量 | 原始输出张量 | CANN 8.0 + PyACL |
| 解码+NMS | decoder | 输出张量 | 边界框 + 类别 + 置信度 | score≥0.5, NMS IOU≤0.5 |
| 跟踪 | deepsort | 检测结果序列 | 轨迹列表（含稳定ID） | max_age=30, min_hits=3 |
| 云台控制 | STM32 | 目标中心坐标 (cx,cy) | 两路 PWM 信号 | PID 参数, 50Hz PWM |

### 2.3 串口通信协议

昇腾端与 STM32 之间的数据交换采用 UART 串口通信，协议设计如下：

```
┌──────┬───────┬───────┬───────┬───────┬──────┬──────┐
│ 帧头 │ cx高位 │ cx低位 │ cy高位 │ cy低位 │ 校验 │ 帧尾 │
│ 0xAA │ 8bit  │ 8bit  │ 8bit  │ 8bit  │ 8bit │ 0x55 │
└──────┴───────┴───────┴───────┴───────┴──────┴──────┘
```

波特率 115200bps，每帧 7 字节。校验和为帧头到 cy 低位所有字节的异或值。STM32 端通过 USART 中断接收，帧头帧尾双校验确保数据完整性。

## 3 关键技术方案

### 3.1 检测前端：MobileNet-SSD

**预处理规范**：模型输入的预处理流程必须与训练阶段的约定保持一致。具体操作为：BGR 转 RGB → resize 到 320×320 → 像素值归一化至 [0,1] → 按 ImageNet 均值与方差标准化 → HWC 转 CHW → 增加 batch 维度。任何环节偏差都会导致推理结果发生明显偏移。

**先验框与解码**：SSD 并非从零生成框，而是在预定义的 default boxes 基础上学习偏移量。网络输出的偏移量需要经过 scale\_xy / scale\_wh 反变换，再从中心点形式转换为左上角和右下角坐标，才能映射回原始图像上的实际边界框。

**NMS 的核心作用**：NMS（非极大值抑制）并不提高分类能力，而是做"去重"——当多个候选框指向同一目标时，只保留最可信的框。这对跟踪环节尤为关键：如果检测器把同一目标输出成多个高重叠框，tracking 模块在创建轨迹时可能会错误地当成多个独立目标，造成重复轨迹和 ID 混乱。

当前工程版本的 NMS 做了效率优化：先对每个 prior 选出一个最佳前景类别，再进行阈值筛选和按类别 NMS，避免在 COCO 等多类别场景里产生大量无效扫描。

### 3.2 跟踪后端：简化版 DeepSORT

**卡尔曼滤波状态估计**：每条轨迹维护一个状态向量 $x = [cx, cy, v_x, v_y]^T$，假设目标在相邻帧之间近似匀速运动。状态转移矩阵 $F$ 为单位阵加上速度项：

$$F = \begin{bmatrix} 1 & 0 & 1 & 0 \\ 0 & 1 & 0 & 1 \\ 0 & 0 & 1 & 0 \\ 0 & 0 & 0 & 1 \end{bmatrix}$$

观测矩阵 $H$ 只观测位置不观测速度。滤波器通过 predict-update 两步平滑观测噪声：先根据上一时刻状态外推当前状态，再用当前检测框修正预测误差。

**匈牙利算法与全局匹配**：当场景中有多条轨迹和多个检测框时，不能只做局部贪心匹配。匈牙利算法从全局角度寻找一一对应的最优分配，避免一个检测框同时分配给多条轨迹。本系统还加入了类别兼容矩阵——如果检测类别与轨迹类别不一致，直接将对应 IOU 置为 -1.0，屏蔽不合理匹配。

**DeepSORT.update 核心流程**：

```
1. 对所有旧轨迹做卡尔曼 predict
2. 构造 IOU 矩阵 + 类别兼容矩阵 → 匈牙利算法求最优匹配
3. 用匹配到的检测结果 update 旧轨迹
4. 对未匹配的检测创建新轨迹
5. 删除 time_since_update > max_age 的轨迹
```

这五步几乎完整体现了多目标跟踪的标准流程。跟踪并不是对检测框简单编号，而是在连续帧之间持续执行预测、匹配、更新与清理。

### 3.3 云台控制：PID 闭环

从跟踪结果中提取目标边界框的中心坐标 $(cx, cy)$，与画面中心 $(320, 240)$ 做差得到偏差信号，PID 控制器据此输出两路舵机的 PWM 调整量：

$$u(t) = K_p \cdot e(t) + K_i \cdot \int_0^t e(\tau)d\tau + K_d \cdot \frac{de(t)}{dt}$$

- **比例项 $K_p$**：与当前偏差成正比，提供即时响应力度
- **积分项 $K_i$**：累积历史偏差，消除稳态误差（如目标静止但不在画面正中心）
- **微分项 $K_d$**：反映偏差变化趋势，在目标快速移动时提前减速，抑制超调和振荡

STM32 端通过 TIM 定时器产生两路 50Hz PWM（脉宽 0.5ms~2.5ms 对应 0°~180°），分别驱动水平轴和俯仰轴的 SG90 舵机。为防止舵机抖动，需要加入输出限幅、死区控制和低通滤波。

## 4 代码结构与二次开发范围

### 4.1 现有代码架构

老师提供的基础代码分为三层：

**入口层**（应用脚本）：
- `scripts/detection_app.py`：纯检测应用，参数解析→选后端→逐帧推理→绘制结果
- `scripts/tracking_app.py`：检测+跟踪应用，在检测基础上接入 DeepSORT

**检测层**（ssdlite 模块）：
- `backend_base.py`：统一检测后端基类，封装预处理、推理调度与输出解码
- `cpu_backend.py`：ONNXRuntime CPU 推理实现
- `npu_backend.py`：Ascend ACL NPU 推理实现
- `decoder.py`：SSD 先验框生成、位置解码、类别概率计算与 NMS

**跟踪层**（tracking 模块）：
- `deepsort.py`：轨迹对象管理、匹配逻辑、轨迹生命周期
- `kalman_filter.py`：卡尔曼滤波状态预测与观测更新
- `utils/postprocessing.py`：检测结果转换为跟踪器输入，并将轨迹画回图像

## 8 风险预案

| 风险场景 | 应对策略 |
|----------|---------|
| 跟踪 ID 频繁跳变 | 优先调整 max\_age（增大容错帧数）、IOU 匹配阈值（放宽关联条件）；使用 --track-classes person 排除其他类别干扰 |
| 基础功能未调稳就急于做云台 | 严格执行分阶段策略，M1 里程碑未达标前不进入第二阶段 |
| 云台跟踪滞后明显 | PID 参数从保守值起步（低 Kp 低 Ki），逐步整定；适当增大串口发送频率 |
| 舵机抖动或过冲 | 加入死区控制（偏差 < 阈值时不响应）、输出限幅、低通滤波平滑 PWM 输出 |
| 串口丢包或数据错位 | 帧头帧尾双校验 + 校验和验证 + 超时重传；极端情况可降帧率保证可靠性 |
| 代码理解不充分 | 三人先各自通读再集体讨论，记录疑问及时向老师确认 |
