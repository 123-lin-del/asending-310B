## 1. 项目概述

### 1.1 项目背景

本项目基于昇腾310B（OrangePi AIpro）边缘计算平台，构建了一套完整的"**检测 → 跟踪 → 云台随动**"闭环系统。USB摄像头采集实时画面，MobileNet-SSD轻量级模型在NPU上推理检测目标，DeepSORT跟踪器维持跨帧目标ID，提取目标中心坐标后通过UART串口发送至STM32F103C8T6，STM32通过CAN总线驱动两台闭环步进电机（Emm_V5.0），使摄像头云台主动跟随目标运动。

### 3.2 端到端数据流

```
第N帧处理流程:
  ① USB摄像头 → OpenCV V4L2 读取帧 (640×480 BGR)
  ② preprocess_frame: BGR→RGB, resize→320×320, ImageNet归一化
  ③ NPU ACL推理: H2D memcpy → mdl.execute → D2H memcpy (~15ms)
  ④ SSDDecoder: 3234个先验框解码 → softmax → 逐类NMS → 筛选(score≥0.35)
  ⑤ DeepSORT: 卡尔曼预测 → IOU矩阵 → 类别兼容 → 匈牙利匹配 → 中心距补充匹配
  ⑥ TargetLock: 锁定最大目标 → 短暂丢失保持 → 丢失超时重新锁定
  ⑦ GimbalSerial: EMA平滑 → 归一化320×240 → 限频20Hz → 串口发送
  ⑧ STM32 UART RX中断: 逐字节接收 → '\n'触发解析 → 双缓冲防竞争
  ⑨ P控制: error_x → rpm_x=Kp×|error| → 死区判定 → CAN发送
  ⑩ 看门狗: 200ms无数据 → 双电机急刹
```

### 4.3 跟踪模块：简化版DeepSORT

**Track类核心属性：**

| 属性 | 说明 |
|------|------|
| `track_id` | 唯一递增ID |
| `bbox` | 当前边界框 `[x1,y1,x2,y2]` |
| `kalman_filter` | 4维卡尔曼滤波器 `[x, y, vx, vy]` |
| `trail` | 最近30个中心点（运动拖尾） |
| `time_since_update` | 未匹配帧计数（超max_age删除） |
| `hits` | 累计匹配次数（低于min_hits不显示） |
| `class_scores` | 历史类别得分累计（投票决定最终类别） |

**跟踪关联流程（DeepSORT.update）：**

```
每一帧:
  ① 所有轨迹做卡尔曼预测（匀速模型 F·x）
  ② 计算轨迹-检测 IOU 矩阵 (N×M, 向量化)
  ③ 计算类别兼容掩码（同类别或任一为-1才允许匹配）
  ④ 匈牙利算法最大化IOU和（scipy linear_sum_assignment）
  ⑤ 阈值过滤：IOU < 0.3 的匹配丢弃
  ⑥ 未匹配的做中心距离补充匹配（归一化距离 < 1.8）
  ⑦ 匹配到的：update → 卡尔曼校正 + 框/分数平滑
  ⑧ 未匹配检测：create_track
  ⑨ 超时轨迹：删除 (time_since_update > max_age)
```

**最终调优参数：**

| 参数 | 默认值 | 最终值 | 调优理由 |
|------|--------|--------|---------|
| `score_threshold` | 0.5 | **0.35** | 降低后检测更多目标，跟踪更稳定 |
| `track_max_age` | 30 | **90** | 增大后轨迹更不容易因短时遮挡断开 |
| `track_min_hits` | 3 | **1** | 减小后新目标立即显示，减少延迟 |
| `track_iou_threshold` | 0.3 | **0.3** | 保持不变，IOU匹配阈值适中 |
| `track_center_distance_threshold` | — | **1.8** | IOU匹配失败后的补充匹配，提升抗遮挡 |
| `track_size_smoothing` | — | **0.8** | 框大小指数平滑，减少跳变 |
| `track_score_smoothing` | — | **0.7** | 分数指数平滑，减少抖动 |

### 4.4 串口输出模块：GimbalSerial + TargetLock

**GimbalSerial核心功能：**

```
从检测结果中挑面积最大的目标 → 计算中心坐标 → EMA低通平滑(smooth=0.4)
→ 归一化至固定虚拟画面320×240 → 按"X{nx}Y{ny}\n"格式发送 → 限频20Hz
```

**TargetLock目标定格功能：**

```
锁定一个track ID持续跟随:
  - 命中锁定目标 → 返回实测框
  - 短暂丢失(≤max_coast帧) → 保持最后位置（不追预测，防跑飞）
  - 丢失超时 → 暂停发送，云台看门狗自动急刹
  - 锁定目标被跟踪器删除 → 清锁，重新锁定当前最大目标
```

**安全机制：**
- 无目标时**不发送**任何数据 → STM端200ms看门狗自动急刹
- 坐标范围自动钳位至`[0, virt_w] × [0, virt_h]`
- 串口写入异常自动禁用，不阻塞主推理循环

### 5.3 主控制逻辑

**P控制器核心参数：**

```c
#define CENTER_X  160         // 虚拟画面中心X (320/2)
#define CENTER_Y  120         // 虚拟画面中心Y (240/2)
#define DEADZONE  25          // 死区（像素），防微抖振荡
float Kp_vel_x = 0.1f;       // X轴比例系数（降低抑制过冲）

#define MAX_RPM_Y  60         // Y轴最大转速（防重力坠机）
#define ACCEL      15         // 加速度（平滑启停）
#define CONTROL_INTERVAL_MS 10 // 控制周期
```

**主循环三部曲：**

```
┌──────────────────────────────────────────────────────┐
│ ① 坐标解析（每收到一帧执行一次）                      │
│   识别帧头'X' → strchr找'Y' → atoi解析坐标           │
│   target_x, target_y → target_valid=1 → 喂狗         │
├──────────────────────────────────────────────────────┤
│ ② P控制器（每10ms执行一次）                           │
│   error_x = target_x - 160                           │
│   if |error_x| > 25: rpm_x = |error_x| × 0.1        │
│   rpm_x = clamp(rpm_x, 0, 150)                       │
│   Vel_Control(1, dir_x, rpm_x, ACCEL, 0)  // CAN发送 │
│   HAL_Delay(1)  ← 错峰，清空CAN邮箱                  │
│   Vel_Control(2, dir_y, rpm_y, ACCEL, 0)             │
├──────────────────────────────────────────────────────┤
│ ③ 看门狗（200ms超时）                                │
│   if 距上次收数据 > 200ms:                           │
│       双电机急刹 → motors_stopped = 1                │
└──────────────────────────────────────────────────────┘
```

### 5.4 关键技术决策

#### 速度环 vs 位置环

| 维度 | 速度环（采用） | 位置环 |
|------|--------------|--------|
| 追踪平滑度 | 连续变速，画面不抖动 | 到达-停-再追，锯齿运动 |
| 目标持续移动 | ✅ 始终保持追赶 | ❌ 永远在"到达-再动"循环 |
| 目标静止 | 接近中心时停转 | 精确定位 |
| 适合场景 | **实时追踪** | 点到点定位 |

**结论**：选择速度环，用死区(DEADZONE=25)防止中心区域来回振荡。

#### CAN应答风暴规避

**问题**：开启CAN RX中断后系统卡死。每个电机命令都会回一帧应答，触发中断风暴（CAN RX中断优先级高于SysTick），导致`HAL_Delay()`永远等不到`HAL_GetTick()`更新。  
**方案**：注释掉`HAL_CAN_ActivateNotification`，本场景只需发送无需接收。CAN RX中断NVIC保留以避免HAL报错，但实际不触发回调。

#### UART双缓冲设计

`rx_buffer`用于实时接收，`process_buffer`供主循环解析。帧结束时`strcpy`拷贝，避免主循环解析过程中被新ISR数据覆盖。

```
                  归一化320×240                  扩展帧 + 分包
```

## 7. 关键技术成果

### 7.1 工程优化汇总

| 序号 | 优化项 | 效果 |
|------|--------|------|
| 1 | V4L2优先 + MJPEG模式 + buffer=1 | 摄像头帧率从~20FPS提升至~26FPS |
| 2 | NMS向量化 + 先验框生成向量化 | 解码阶段CPU开销显著降低 |
| 3 | IOU矩阵向量化计算（替代双层循环） | 匹配阶段更快 |
| 4 | 解码阶段按指定类别过滤 | 减少不关心类别的后处理开销 |
| 5 | EMA低通滤波平滑串口坐标 | 减少逐帧抖动，云台运动更平滑 |
| 6 | TargetLock目标定格 | 避免"人走快了跟到别的目标上" |
| 7 | CAN中断风暴规避 | 系统稳定运行，不再卡死 |
| 8 | 双缓冲UART接收 | 避免解析竞争，帧同步可靠 |
| 9 | 看门狗安全机制 | 目标丢失自动急刹，安全可靠 |
| 10 | Y轴限速+死区防振荡 | 云台画面稳定不抖动 |

### 7.2 算法理解成果

| 算法 | 核心理解 |
|------|---------|
| **MobileNet** | 深度可分离卷积 = Depthwise + Pointwise，参数量大幅降低，适合边缘设备 |
| **SSD** | 一阶段检测器，多尺度特征图+先验框直接回归，NMS去重 |
| **卡尔曼滤波** | Predict(匀速模型外推) + Update(观测修正)，卡尔曼增益自适应权衡 |
| **匈牙利算法** | 全局最优一一匹配，最大化IOU和，优于局部贪心 |
| **P控制器** | 误差比例驱动转速，速度环平滑追踪，死区防振荡 |

### 7.3 问题解决记录

| 问题 | 影响 | 解决方案 |
|------|------|---------|
| CAN应答中断风暴 | 系统卡死，第二条命令发不出 | 关闭CAN RX中断，只发不收 |
| 摄像头实时帧率低 | 检测链路FPS仅~20 | V4L2优先 + MJPEG + buffer=1 → ~26 FPS |
| Y轴电机重力下坠 | 断电或停止时云台低头 | MAX_RPM_Y=60限速 + 看门狗急刹 |
| 目标在中心附近电机振荡 | 云台抖动，画面不稳 | DEADZONE从15增大至25 |
| 跟踪ID频繁切换 | 同一目标被识别为新ID | max_age 30→90, min_hits 3→1, 补充匹配 |
| 快速目标IOU匹配失败 | 轨迹断开 | 中心距离补充匹配 threshold=1.8 |
